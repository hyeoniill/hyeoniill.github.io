## Git을 왜 사용하는가?

Git


### 원격 <> 로컬

> 원격: 깃 저장소에 저장된 코드
> 로컬: 내 컴퓨터에 저장된 코드

깃을 사용한 협업을 이해하기 위해 가장 기초가 되는 개념이다.  
원격은 깃의 저장소에 저장되어 있는 코드로써 협업을 하는 사용자가 함께 사용하고 수정할 수 있다.  
반면 로컬은 내가 실제 작업을 하는 코드로 사용자가 개발을 할 수 있도록 한다.

원격은 본체, 로컬은 복사본의 개념으로 생각하면 된다. 복사본의 코드는 아무리 수정해도 원문에는 영향이 가지 않고, 복사본의 코드가 올바른 것을 확인한 후에 원문을 업데이트 할 수 있다.

### Git hash
해시값으로 커밋을 변경하려할 때는 구분이 가능하면 됨

## Git 커밋

> VCS: Version Control System

Git은 VCS 시스템으로써 코드를 누가, 언제, 무엇을 바꾸었는지 기록해주는 시스템이다. 
각각의 변경 이력을 커밋을 통해서 관리한다. 그렇기 때문에 깃을 통햇 버전관리가 가능해진다.
커밋은 내부적으로 해시값을 갖고, 변경된 코드 이력을 해시 값을 통해서 관리한다.

명령어
> pwd: print working directory 현재 작업 디렉토리 경로 출력  
> ls -al:   
> git init: 현제 디렉토리를 로컬 레포지토리로 만듬


git commit --amend : 
가장 최신 커밋의 제목을 변경하는 옵션, 커밋의 해시값을 변경한다.  
로컬에서는 사용해도 충돌이 발생하지 않으나 원격에 있는 커밋을 --amend로 변경하게 되면 동기화되지 않는 문제가 발생한다.  
그렇기 때문에 원격에 있는 커밋을 --amend로 변경하는 것은 **민폐**임.

git diff: 변경된 코드의 세부 내용을 확인할 수 있음

> git diff: add되지 않은 내용과 커밋간의 변경점  
> git diff --staged: add된 내용과 커밋간의 차이  


## 브랜치

### merge <> rebase 차이점
git merge: merge하는 브랜치의 커밋을 그대로 사용하여 그래프를 나누어 사용
git rebase: rebase하는 브랜치에서 변경된 내용에 대한 커밋을 새로 작성하여 rebase의 커밋으로 작성하고 그래프를 하나로 유지


### git stash
스택 구조로 동작

## 브랜치 전략

## git reset

### git reset --soft
커밋을 지우고, 스테이지는 유지

### git reset --mixed
커밋을 지우고, 스테이지도 해제

### git reset --hard
커밋을 지우고, 수정 기록도 삭제 = 해당 커밋으로 초기화
