해시테이블은 Key와 Value를 한 쌍으로 저장하는 자료구조이다.
해시 함수가 키를 버킷 인덱스로 바꾼 뒤, 그 위치(또는 충돌 시 연결된 구조)에 데이터를 둔다.

실제 구현에서는 **로드 팩터**(저장된 원소 수 ÷ 버킷 수에 가까운 지표, 자바 `HashMap`은 기본 0.75 근처에서 테이블 확장)에 따라 테이블을 키우고 **재해싱**하기도 한다.
시간 복잡도는 **평균적으로** 삽입·조회·삭제가 O(1)에 가깝고, 최악(모든 키가 같은 버킷으로 몰리는 등)에는 O(n)에 가까워질 수 있다.

## Direct-address Table

<!-- TODO: Direct-address — 유니버스 크기 U vs 실제 키 개수 n, 공간 O(U) -->

```java
student[0] = "Alice";
student[1] = "Bob";
```

키가 작은 정수이고 **가능한 키의 범위(유니버스)**가 좁을 때는 배열 인덱스로 바로 접근할 수 있다.
다만 테이블 크기는 보통 **유니버스 크기 U**에 비례하므로, 실제로 쓰는 키 개수 n이 U보다 훨씬 작으면 공간 낭비가 크다.

키 값이 0부터 시작하지 않고 20260512부터 시작한다고 했을 때, 그 이전의 공간에는 저장하지 않기 때문에 메모리 낭비가 발생한다.
또한 키 값으로 "st20260512"와 같은 문자열이 입력되었을 경우에도 저장할 수 없는 문제가 있다.

## 해시 테이블

<!-- TODO: 해시 테이블 섹션 — 해시함수 h(k), 인덱스 = h(k) mod m, 왜 direct address보다 나은지 -->

직접 주소 방식 대신 **해시 함수** h를 써서 키 k를 정수 해시값으로 만든 뒤, 버킷 개수 m에 대해 `h(k) mod m` 같은 방식으로 **제한된 크기의 배열**에 매핑한다.
이렇게 하면 키가 문자열·객체 등이어도 동일한 틀로 다룰 수 있고, 테이블 크기는 **실제 저장량과 로드 팩터**에 맞춰 조절할 수 있다.

## Collision

<!-- TODO: Collision — separate chaining vs open addressing; Java HashMap은 chaining + (많으면) tree -->

서로 다른 두 키가 같은 버킷 인덱스로 가는 경우를 **충돌(collision)**이라 한다. 이론적으로 완전히 피할 수는 없으므로 **해결 전략**이 필요하다.

- **분리 연결(separate chaining)**: 버킷마다 리스트(또는 트리)를 두고 같은 인덱스의 항목을 묶는다. 자바 `HashMap`이 이 계열이다.
- **개방 주소(open addressing)**: 같은 배열 안에서 다른 빈 칸을 찾아 저장한다(선형 탐사, 이차 탐사, 이중 해싱 등).

## Hash Map

<!-- TODO: HashMap — Java 8+ 버킷: list → tree 전환 임계, O(1)은 평균 -->

해시맵도 기본적으로 배열[Array]의 형태이다.
배열 안에는 key, value로 이루어진 Node가 저장된다.

키 값은 유일[Unique]해야 하기 때문에 기존에 있던 키와 동일한 키 값으로 put을 한다면 새로운 값으로 대체되게 된다.

다만 다른 키에 대해서 동일한 인덱스가 발생하면, 해당 버킷에서 **충돌을 처리**한다. JDK에서는 버킷당 항목이 적을 때는 **연결 리스트**로 두고, 같은 버킷에 쌓이는 원소가 많아지면(JDK 8 이후) **균형 트리**로 바꿔 최악 경우의 탐색 비용을 줄이는 방식을 쓴다.

해시맵의 장점은 키만 주어지면 대응하는 값을 **평균적으로** O(1)에 가깝게 찾을 수 있다는 점이다.
또한 해당 키가 존재하는지 여부도 **평균적으로** O(1)에 가깝다.

**해시·동등성 관점에서는 Value는 무관하다.** 키의 `hashCode()`와 `equals()`만으로 버킷과 동일 키 여부가 결정되며, 값은 그 키에 매달리는 페이로드일 뿐이다.

## 해시맵/해시셋 크기

<!-- TODO: 용량 설명 수정 — initial capacity vs size, load factor 0.75, 리사이즈 시점 -->

<!-- TODO: 62행 — "빈 공간" 표현을 로드 팩터/리사이즈 맥락으로 정정 -->

해시맵은 내부적으로 연속된 메모리 공간인 배열에 저장된다.
자바에서는 `new HashMap<>()`처럼 **초기 용량을 생략**할 수 있다.

```java
new HashMap<>();
```

내부 배열 길이는 **초기 용량(initial capacity)**과 확장 규칙으로 정해진다.
기본 초기 용량은 16이며, 구현상 테이블 길이는 **2의 거듭제곱**으로 맞추는 경우가 많다.
`hash & (n - 1)` 형태로 인덱스를 구할 수 있어 **비트 연산**으로 나머지 연산을 대신하기 쉽다.

```java
new HashMap<>(100);
```

생성자에 넘기는 값은 **“원하는 초기 용량”**이지, 현재 원소 개수(size)와 같다는 뜻이 아니다.
예를 들어 초기 용량 100이면 내부 테이블은 그보다 작지 않은 **2의 거듭제곱**(예: 128)으로 잡힐 수 있다.

원소가 늘어나 **로드 팩터**(기본 0.75)를 넘기면 테이블이 커지고 **재해싱**이 일어난다.
충돌이 덜 “느껴지게” 하려면 단순히 “빈 칸이 많다”기보다, **버킷 수 대비 원소 수(로드 팩터)**를 낮게 유지하는 쪽이 본질에 가깝다.
테이블이 과밀해지면 같은 버킷에 몰리기 쉬워져 탐색·삽입 비용이 나빠질 수 있다.

### HashMap 순회

해시맵은 저장한 순서를 보장하지 않는다.

## HashSet

<!-- TODO: HashSet — HashMap 백킹, equals/hashCode로 유일성 -->

해시셋은 중복된 요소를 허용하지 않는 집합을 구현한 자료구조이다.
OpenJDK 계열에서는 내부적으로 **`HashMap`을 백킹**해 두고, 집합 원소를 키처럼 두는 방식으로 구현되는 경우가 일반적이다.
중복 여부는 원소의 `hashCode()`와 `equals()`로 판단한다.
